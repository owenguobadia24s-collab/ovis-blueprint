# ADR-007 — Canonical Identity and ID Generation

## Status
Accepted

## Date
2026-03-15

## Owner
OVIS

---

## Context

ADR-001 established the canonical OVIS object model, lineage contract, and append-only audit requirements.

ADR-002 established function calling as the canonical capability bus and fixed schema, idempotency, and result-envelope discipline as governance requirements for execution.

ADR-003 established the Responses API as the canonical runtime substrate and the runtime wrapper as the only canonical model invocation path.

ADR-004 established ZDR-first default session posture and explicit continuation handling without transferring authority to retained runtime state.

ADR-005 established the review gate and policy profiles so that execution authority remains governed as work transitions from planning to execution.

ADR-006 established the canonical `Branch` object as the continuity container and the `Compaction Record` as the governed continuity-reduction artifact.

What remains insufficiently fixed is the canonical identity model itself, including:

- how canonical IDs are generated
- how causal identity is propagated
- how event, branch, compaction, and job identities remain stable over time
- how runtime continuity identifiers relate to canonical OVIS identity
- how identity is kept distinct from sequence ordering and content hashing

The current workspace and scaffolds already depend on canonical IDs, but without a fixed identity decision they remain vulnerable to:

- identity drift across packages and scripts
- ambiguous audit trails
- inconsistent causality propagation
- accidental reuse of content hashes as identities
- sequence-like numbering being mistaken for canonical identity

Therefore OVIS requires a canonical identity and ID generation decision that future implementation work can rely on without reinterpreting identity semantics at each layer.

---

## Decision

OVIS shall use OVIS-owned canonical IDs for governed identity across runtime, event logging, branch continuity, compaction, and job artifacts.

Canonical IDs are governance objects, not convenience strings.

They exist to preserve:

- stable auditability
- causal continuity
- lineage reconstruction
- cross-layer object reference integrity
- separation of identity from content representation and ordering

OVIS canonical IDs shall be:

- OVIS-owned
- explicitly typed by identity family
- stable once issued
- distinct from content hashes
- distinct from sequence indexes
- validated at canonical boundaries

Correlation IDs are propagated across a causal flow.

They are not regenerated mid-flow.

Sequence ordering may exist for display, storage, or processing convenience, but it must not be conflated with canonical identity.

Content hashes may exist for integrity or deduplication purposes, but they must not replace canonical IDs.

---

## Canonical Identity Principles

The canonical OVIS identity model shall follow these rules:

1. every identity family must have an explicit purpose and scope
2. every canonical ID must be generated by an OVIS-controlled mechanism rather than by ad hoc script convention
3. every canonical ID must have a type-distinguishable format
4. every canonical ID must remain stable for the lifetime of the governed identity it names
5. identity propagation rules must preserve causality without redefining object authority
6. deterministic derivation is prohibited unless explicitly allowed for that identity family
7. content hashes, transcript spans, and sequence positions may supplement identity, but they do not replace it

Canonical identity therefore belongs to OVIS governance, not to storage backend accident, runtime retention state, or content-addressing convenience.

---

## Canonical Format Direction

The canonical format direction for OVIS IDs is:

- typed prefix
- delimiter
- opaque unique suffix

The canonical suffix form for the identity families defined here is a lowercase UUID string.

This means the default canonical format is:

- `<prefix>_<uuid>`

Examples of canonical family prefixes include:

- `evt_`
- `br_`
- `corr_`
- `cmp_`
- `job_`

The purpose of the prefix is to:

- make identity family explicit
- prevent cross-type confusion
- support validation and review tooling
- reduce accidental misuse across runtime, audit, and continuity boundaries

The purpose of the opaque suffix is to:

- provide uniqueness without semantic overloading
- avoid embedding sequence claims into identity
- avoid making content-derived identity the default

No implementation may treat lexical ordering of the suffix as canonical sequence meaning.

## event_id

### Purpose

`event_id` names one canonical append-only `Event`.

Its purpose is to:

- identify a single audit event uniquely
- support parent-event linkage
- support correlation-aware audit reconstruction
- preserve append-only identity even when payloads are corrected or superseded by later Events

### Scope

`event_id` scope is one canonical `Event` record.

It must not be reused across:

- different audit events
- retries that produce distinct append-only Events
- child events related to the same causal chain

### Generation Rules

`event_id` shall be generated at the moment a canonical Event is created for append-only emission.

It shall be generated by an OVIS-controlled emitter, wrapper, or governed helper.

It shall not be derived from:

- event payload text
- event position in a sequence
- storage row order
- content hash

### Format

Canonical format:

- `evt_<uuid>`

### Uniqueness Expectations

Each canonical Event must have a unique `event_id`.

Two distinct append-only Events must never share the same `event_id`, even if their payloads are identical.

### Determinism vs Non-Determinism

`event_id` is non-deterministic by default.

It must not be deterministically recomputed from event content.

### Propagation Rules

`event_id` is not propagated across a flow.

Instead:

- `parent_event_id` links related Events
- `correlation_id` preserves shared causal grouping

### Validation Expectations

Validation must confirm:

- `event_id` is present
- `event_id` matches the `evt_` family format
- `event_id` is not reused for a distinct Event
- `event_id` is not confused with `correlation_id` or `parent_event_id`

---

## branch_id

### Purpose

`branch_id` names one canonical `Branch` continuity container.

Its purpose is to:

- preserve recursive continuity identity
- associate governed work artifacts to one continuity container
- survive runtime session rotation, expiration, and compaction
- anchor downstream planning, execution, and audit to one branch lineage

### Scope

`branch_id` scope is one governed branch of work.

It persists across:

- multiple Signals updating the same branch
- multiple runtime turns
- multiple Events within the branch
- multiple compactions of the same branch

### Generation Rules

`branch_id` shall be generated once at canonical branch creation.

It shall not be regenerated because:

- runtime sessions change
- continuation identifiers change
- compaction occurs
- branch state is reduced or archived

### Format

Canonical format:

- `br_<uuid>`

### Uniqueness Expectations

Each canonical Branch must have one unique `branch_id`.

No two unrelated branches may share the same `branch_id`.

### Determinism vs Non-Determinism

`branch_id` is non-deterministic by default.

It must not be derived from:

- signal content
- branch title
- sequence number
- storage path

Deterministic branch import or migration rules, if ever needed, require a later explicit decision.

### Propagation Rules

`branch_id` is propagated across all branch-linked objects and Events where branch continuity is material.

It is preserved through:

- runtime continuation
- planning
- execution support
- compaction
- audit emission

### Validation Expectations

Validation must confirm:

- `branch_id` is present
- `branch_id` matches the `br_` family format
- branch-linked objects do not silently substitute runtime session identifiers for `branch_id`
- compaction does not replace or mutate `branch_id`

---

## correlation_id

### Purpose

`correlation_id` names one causal flow or governed action chain.

Its purpose is to:

- preserve causality across runtime, capability execution, event emission, and compaction-related operations
- group related Events without collapsing them into one identity
- support replay, audit reconstruction, and idempotency reasoning

### Scope

`correlation_id` scope is one causal flow, not one object type.

It may span:

- runtime turns
- function-called capabilities
- event emission related to one governed operation
- compaction operations and their resulting Events

### Generation Rules

`correlation_id` shall be generated once at the start of a governed causal flow.

It shall be issued by an OVIS-controlled runtime, command, or orchestration boundary.

It shall not be generated independently by downstream scripts once a flow already has a canonical `correlation_id`.

### Format

Canonical format:

- `corr_<uuid>`

### Uniqueness Expectations

Each active governed causal flow must have one `correlation_id`.

Different causal flows should use different `correlation_id` values even when they affect the same `branch_id`.

### Determinism vs Non-Determinism

`correlation_id` is non-deterministic at causal-flow creation.

After creation, it becomes propagation-stable rather than re-generated.

It must not be deterministically recomputed from payload or sequence state.

### Propagation Rules

`correlation_id` is propagated, not regenerated, across a causal flow.

This means:

- runtime continuation must preserve it
- function-called execution must preserve it
- related Events must preserve it
- compaction-related operations must preserve it where they belong to the same causal flow

A new `correlation_id` may be created only when a genuinely new causal flow begins.

It must not be regenerated mid-flow for convenience.

### Validation Expectations

Validation must confirm:

- `correlation_id` is present where causal grouping is required
- `correlation_id` matches the `corr_` family format
- child operations do not silently replace the inherited `correlation_id`
- parent-child Events in one causal chain preserve correlation continuity unless an explicit later decision defines a fork rule

## compaction_id

### Purpose

`compaction_id` names one canonical `Compaction Record`.

Its purpose is to:

- identify one governed compaction operation
- link compacted state to the branch and preserved references it represents
- support append-only compaction audit events
- distinguish multiple compactions of the same branch from one another

### Scope

`compaction_id` scope is one compaction operation and its resulting `Compaction Record`.

It does not replace:

- `branch_id`
- `event_id`
- `correlation_id`

### Generation Rules

`compaction_id` shall be generated when a canonical compaction operation creates a `Compaction Record`.

It shall be OVIS-issued and shall not be derived from:

- branch sequence count
- compacted payload text
- summary text
- content hash

### Format

Canonical format:

- `cmp_<uuid>`

### Uniqueness Expectations

Each compaction operation must have a unique `compaction_id`.

Multiple compactions of the same branch must each receive distinct `compaction_id` values.

### Determinism vs Non-Determinism

`compaction_id` is non-deterministic by default.

It must not be recomputed deterministically from the same source range or the same compacted output.

### Propagation Rules

`compaction_id` is propagated to:

- the canonical `Compaction Record`
- Events that materially refer to the compaction artifact
- branch state where `latest_compaction_id` or equivalent canonical linkage is required

It is not a replacement for `branch_id`.

### Validation Expectations

Validation must confirm:

- `compaction_id` is present
- `compaction_id` matches the `cmp_` family format
- branch compaction surfaces do not reuse one `compaction_id` for multiple compaction operations
- compaction linkage does not confuse `compaction_id` with `branch_id`

## job_id

### Purpose

`job_id` defines the canonical identity family for OVIS job-class objects.

Its purpose is to:

- provide one common identity rule for governed job artifacts
- preserve job identity across planning and execution layers
- support audit, review, and execution references without sequence-based naming

### Scope

`job_id` is the family rule underlying concrete job identifiers such as:

- `plan_job_id`
- `execute_job_id`

ADR-001 remains authoritative for the concrete canonical object fields.

ADR-007 defines the generation and validation rule for the job identity family those fields belong to.

### Generation Rules

Each canonical job artifact shall receive an OVIS-issued job-family identity at creation.

Concrete job object fields shall use family-specific typed prefixes.

This means:

- a `Plan Job` receives a `plan_job_id`
- an `Execute Job` receives an `execute_job_id`
- both remain part of the canonical job identity family

Job-family IDs shall not be derived from:

- work-object sequence position
- plan ordering
- execution retry count
- storage row order

### Format

Canonical job-family format direction:

- `job_<job_class>_<uuid>`

Concrete canonical field formats are:

- `job_plan_<uuid>`
- `job_execute_<uuid>`

### Uniqueness Expectations

Each canonical job artifact must have a unique job-family ID.

No two distinct `Plan Job` or `Execute Job` objects may share the same concrete job ID.

### Determinism vs Non-Determinism

Job-family IDs are non-deterministic by default.

They must not be recomputed from:

- work-object identity
- previous job text
- retry state
- content hash

### Propagation Rules

Concrete job IDs are propagated to:

- review-gate decisions where material
- execution transitions
- Events referring to the job artifact
- downstream canonical references that point to the job object

They are not replaced by:

- sequence counters
- retry numbers
- result hashes

### Validation Expectations

Validation must confirm:

- concrete job IDs are present on canonical job objects
- concrete job IDs match the expected job-family prefix
- `plan_job_id` and `execute_job_id` are not used interchangeably
- job identity is not inferred from ordering or content-derived surrogates

## Identity, Sequence Ordering, and Content Hashes

OVIS canonical identity must remain separate from both sequence ordering and content-addressing.

This means:

- sequence numbers may be used for display or processing order
- sequence numbers do not become canonical identity
- content hashes may be used for integrity, deduplication, or payload anchoring
- content hashes do not become canonical identity

Canonical IDs answer:

- what governed identity is this

Sequence indexes answer:

- in what order was this observed or processed

Content hashes answer:

- what content fingerprint is associated with this payload

These are different governance concerns and must not be collapsed into one field.

## Relationship to ADR-001

ADR-001 defines the canonical OVIS objects and requires canonical IDs as lineage anchors.

ADR-007 fixes how those identity anchors are generated, propagated, and validated so that canonical object references remain stable across runtime, review, execution, audit, branch continuity, and compaction.

---

## Relationship to ADR-002

ADR-002 requires idempotency, structured execution boundaries, and canonical result envelopes.

ADR-007 supports that by fixing:

- stable job-family identity
- stable correlation propagation
- canonical event identity distinct from retries and result payloads

Function-called execution must therefore preserve canonical IDs rather than invent local substitutes.

---

## Relationship to ADR-003

ADR-003 fixes the runtime wrapper as the only canonical model invocation path.

ADR-007 requires that runtime behavior preserve canonical identity rather than allowing provider response IDs, session handles, or script-local request identifiers to become sovereign OVIS identity.

Provider-managed IDs may exist as external references, but they do not replace canonical OVIS IDs.

---

## Relationship to ADR-004

ADR-004 requires explicit continuation and OVIS-owned branch continuity under a ZDR-first posture.

ADR-007 preserves that by ensuring:

- `branch_id` remains the continuity anchor
- `correlation_id` remains propagation-stable across a causal flow
- runtime continuation identifiers do not replace canonical branch or causal identity

---

## Relationship to ADR-005

ADR-005 defines the review gate and policy profiles as the canonical execution-governance boundary.

ADR-007 ensures that approvals, planning, and execution can preserve stable job and correlation identity across that boundary without relying on sequence markers or content-derived surrogates.

---

## Relationship to ADR-006

ADR-006 defines canonical branch continuity and compaction.

ADR-007 preserves that by fixing:

- `branch_id` as the continuity identity
- `compaction_id` as the governed identity of one compaction operation
- `correlation_id` as the causal identity carried through related compaction flows

Compaction therefore preserves identity-bearing continuity rather than replacing identity with opaque reduced state.

---

## Immediate Consequences

1. OVIS now has a canonical identity model for `event_id`, `branch_id`, `correlation_id`, `compaction_id`, and the job identity family.
2. Future CJ-006 through CJ-012 work must generate canonical IDs through OVIS-controlled helpers rather than ad hoc local strings.
3. Correlation IDs must be propagated across a causal flow rather than regenerated mid-flow.
4. Sequence ordering and content hashes must remain separate from canonical identity.
5. Provider-managed or storage-managed identifiers may exist as references, but they do not replace canonical OVIS IDs.
6. Validation surfaces must enforce prefix, format, scope, and family-correctness for canonical IDs at governed boundaries.

---

## Deferred to Follow-on Decisions

The following are intentionally deferred:

- exact implementation helper package for canonical ID issuance
- exact migration and import rules for legacy objects that predate canonical ID issuance
- exact relationship between canonical IDs and any future human-friendly display identifiers
- exact handling of foreign external IDs when bridged systems must be referenced canonically
- exact operational policy for backfilling canonical IDs into transitional storage layers

These will be defined in dependent ADRs and implementation specifications.

---

## Summary

OVIS cannot preserve stable lineage, auditability, causality, or continuity if identity is left to storage accident, sequence convention, or content-derived surrogates.

This ADR establishes canonical OVIS-owned identity for events, branches, causal flows, compactions, and job artifacts, fixes generation and propagation rules for those identity families, and requires that future implementation preserve the distinction between canonical IDs, sequence ordering, and content hashes across ADR-001 through ADR-006.
